# 1. Project Architecture

: FSD(Feature-Sliced-Design)폴더 구조

## FSD 아키텍처 선택 이유

해당 프로젝트는 AI활용 능력과 다른 대규모 프로젝트 내 FSD 아키텍처를 도입하기 전 연습 및 테스트용으로 적용해보기로 하였다.

## FSD 아키텍처란?

SW 구조를 기능 중심으로 나누어 모듈화하고, 각 기능을 독립적으로 관리하는 방식
이 아키택처는 대규모 프론트엔드 어플리케이션에서 코드의 유지보수성과 재사용성을 높이기 위해 사용된다.
각 기능을 독립적인 도메인으로 분리하여 개발할 수 있도록 하고, 팀 협업 환경을 개선하며, 코드의 복잡성을 낮추기 위한 목표를 가진다.

## FSD 아키텍처의 기본 개념

FSD는 기능 중심(Feature-Oriented) 접근 방식을 사용한다.
이로 인해 프로젝트가 점진적 성장 환경을 가져도 유지보수가 용이하게 설계가 된다.
기본 원칙은 아래와 같다.

1. 기능 기반 분할 : 어플리케이션을 기능별로 나누어 각 기능을 독립적인 모듈로 관리하며, 기능은 시스템에서 특정한 비즈니스 로직이나 사용자 인터페이스의 한 부분을 담당한다.
2. 레이어링 : 기능을 레이어(Layer)로 나누어 책임을 분리한다. 일반적으로 7개의 레이어로 구성된다.(app, pages, widget, features, entities, shared, process(선택)) 레이어는 단방향(상위-> 하위)성을 가지며, 위에서 아래로만 의존이 가능하다.
3. 폴더 구조 : 기능별로 폴더 구조를 구성하여 코드의 가독성과 유지보수성을 높이며, 각 폴더는 그 기능과 관련된 모든 것을 포함한다.
4. 재사용성 : 공통 기능이나 유틸리티는 재사용 가능하도록 shared 레이어에 배치한다. 이를 통해 중복 코드를 줄이고 일관성을 유지한다.

## Feature Slice 분리 기준

해당 프로젝트는 동사 기준으로 기능들을 분리한다면 아래와 같은 동사들이 도출되었다.

1. create(생성)
2. delete(삭제)
3. set(설정)
4. edit(수정)
5. check(체크)
   그리고 5번을 제외한 모든 동사는 2개의 목적어를 가졌다. 정리해본다면 아래와 같다.

- create account, create post
- delete account, delete post
- set post, set user
- edit post, edit user
- check attendance
  이러한 이유로 feature 레이어의 하위 폴더는 동사 단위로 분리하되, 1개의 목적어를 가진 요소는 편의성을 위해 동사와 목적어를 합친 하위 폴더명을 설정하였다.

```
features/
├── create/
|   ├── account/  : 계정 생성
|   └── post/     : 포스트 생성
├── delete/
|   ├── account/  : 계정 삭제
|   └── post/     : 포스트 삭제
├── set/
|   ├── post/     : 포스트 설정
|   └── user/     : 유저 설정
├── edit
|   ├── post/     : 포스트 수정
|   └── user/     : 유저 수정
└── checkAttendance/ : 출석 체크
```
